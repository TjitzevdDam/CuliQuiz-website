User story

The problem we solve

Building a training is work: thinking up questions, checking answers, assembling content. That is the barrier that holds new customers back. Menu AI removes it.

As a hospitality operator I want to upload my menu, so my team trains on our own dishes within minutes, without me creating questions by hand.
The definition of done is below, testable and numbered. This is the core: take the work away.
The user flow

Five steps, with input and output

Step 1 is done by the user. Steps 2 to 4 are Menu AI. Step 5 is the user’s choice. Each step shows what goes in and what comes out.

1

Upload

In: food / drinks menu (PDF or photo, multiple files allowed).
Out: a stored source document at the organisation.

2

Recognise

In: the document.
Out: per item dish/drink, ingredients, price, composition. Inspectable.

3

Generate

In: the recognised items.
Out: per item all fitting questions, linked to the Knowledge Graph.

4

Validate

In: the generated set (status "pending").
Out: a Content Set approved by CuliQuiz, ready to use.

5

Offer

In: the approved Content Set.
Out: a campaign, tournament or battle, chosen by the user.

How we fill games

Filter the database, pull the questions

A game or campaign is a selection from the database. Filter by organisation (the own menu) and sub-topic (food, wine, drinks…), and the linked questions fill the game automatically.

Database→ Filter: org + sub-topic→ Game / Content Set

Every uploaded menu adds entities and questions. The more menus, the richer the database, for the whole platform.

// data model (simplified)
Entity { id, type: ingredient | dish | drink, name, organization_id, subtopic: food | wine | drinks | …, allergens, price }
   └─ has many → Question { game_type, prompt, correct, explanation, … }

Acceptance criteria

Definition of done

Testable. Only when all points hold is Menu AI phase 1 done. The three constraints at the top are built in here.

  1. Upload accepts PDF and photo, also multiple files at once and scans without a text layer (OCR).
    A failed or unreadable upload gives a clear message, no silent error.
  2. Extraction delivers per item dish/drink, ingredients, price and composition, and is inspectable per item before generation.
  3. Per item all fitting game types generated constraint
    At least True/False + MCQ where applicable; sorting/image where the item lends itself to it.
  4. Every question is linked to domain / subdomain / topic from the Master Knowledge Graph and carries source_type=organization. No loose questions.
  5. Output is a Content Set in platform format, directly selectable in the campaign builder, without manual rework.
  6. Nothing live without approval constraint
    Every set starts at "pending_validation" and is reviewed by the CuliQuiz content team before publication.
  7. To a playable set within ~5 minutes constraint
    Asynchronous processing with status or notification is allowed; the user does not wait on a blank screen.
  8. Distribution choice when offering: campaign, tournament or battle.
  9. Entity database filled: every item is stored as an entity (linked to organisation + sub-topic) with its question variants; games are filled from it. Enriches the global pool (dedup, see open items).
Distribution

The user picks the format

The same approved content, three ways to roll it out.

Fixed campaign

Continuous programme with duration and goals. For structural knowledge building.

Tournament

Temporary competition with a leaderboard. For energy and a peak moment.

Battle

Fast, head-to-head. For instant adoption: play right away, learn right away.
